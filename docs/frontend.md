# Frontend

The frontend is a React 18 + TypeScript application built with Vite. It runs inside the macOS Webview (WKWebView) and communicates with the Go backend exclusively through the Wails-generated bindings in `frontend/wailsjs/`.

---

## Entry Points

| File | Role |
|---|---|
| `frontend/index.html` | HTML shell; mounts `#app` |
| `frontend/src/main.tsx` | React root; wraps `<App>` in all context providers |
| `frontend/src/App.tsx` | React Router setup; renders the `<Layout>` with all routes |

│  TopBar (title, language switcher, OAuth dropdown)       │

`Layout.tsx` composes `TopBar`, `Sidebar`, and `ContentArea`. All three share context values via their respective context providers.

| Context | File | Provides |
|---|---|---|
| `SidebarContext` | `Layout/Sidebar/SidebarContext.tsx` | Collapsed state of sidebar |
| `MenuContext` | `Menu/MenuContext.tsx` | Active menu items |
| `TopBarContext` | `TopBar/TopBarContext.tsx` | Page title |

| `LanguageContext` | `LanguageSwitcher/LanguageContext.tsx` | Active language |
| `MessageTypeContext` | `Notification/MessageTypeContext.tsx` | Toast message queue |

## Routing

Routes are built dynamically in `config/RouteBuilder.tsx`. Static routes (playlists, settings, etc.) are defined there; plugin routes are appended at runtime after plugins are loaded.

Plugin routes are discovered via `config/plugins.ts`:

1. Fetch `GET /api/plugins` → list of installed plugins.
2. For each plugin, fetch its `manifest.json`.
3. Dynamically `import()` the plugin's `index.js` from the callback server.
4. Extract `pluginRoute` and `pluginMenu` exports.
5. Register in `RouteBuilder` and `MenuBuilder`.

---

## Wails Bindings

`frontend/wailsjs/go/main/App.js` and its TypeScript declaration `App.d.ts` are auto-generated by Wails from the public methods of `*App` in Go. These are the only way the frontend calls Go functions.

`frontend/wailsjs/runtime/runtime.js` provides Wails runtime utilities: event subscription, dialog openers, etc.

## Music Player (`components/Music/`)

```
Music/
├── Music.tsx                    # Route page: playlist list + actions
├── Playlist/
│   ├── Playlist.tsx             # Playlist detail view
│   ├── View/GridView.tsx        # Album-art grid layout
│   └── View/ListView.tsx        # Table list layout
└── Player/
    ├── MusicPlayer.tsx          # Audio element controller
    ├── PlayerContext.tsx        # Global playback state
    ├── Playlist.tsx             # Now-playing queue
    └── Skin/
        ├── Ava.tsx              # Full player skin
        ├── AvaMinimal.tsx       # Compact skin
        └── MinimalBar.tsx       # Mini bottom bar
```

The player uses the native HTML `<audio>` element. Tracks are loaded as base64 data-URLs via `GetAudioDataUrl()` to avoid Webview cross-origin restrictions with local `file://` paths.

## Internationalisation

Three locales are supported: **English** (`en`), **German** (`de`), **Italian** (`it`).

Translation files live at `frontend/public/locales/<lang>/translation.json`. They are loaded at runtime by `i18next-http-backend`. The active language is stored in `LanguageContext` and applied via the `useTranslation` hook.

## UI Components (`components/ui/`)

Primitive components that are reused across the app:

- **Buttons**: `BlueButton`, `CreateButton`, `EditButton`, `DeleteButton`, `CancelButton`, `GridViewModeButton`, `ListViewModeButton`
- **Alerts**: `ErrorAlert`, `InfoAlert`, `SuccessAlert`, `WarningAlert`, `NeutralAlert`
- **Pills**: `CreatePill`, `DeletePill`, `DisabledPill`
- **Dropdown**: generic dropdown
- **button.tsx**: base button with `class-variance-authority` variants

---

## Styling

Tailwind CSS 3 with the `tailwindcss-animate` plugin. Class merging uses `clsx` + `tailwind-merge` via the `cn()` utility in `lib/utils.ts`. Component variants are defined with `class-variance-authority`.
